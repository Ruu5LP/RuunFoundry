# プロジェクト規約

## ファイル構成規約

- `src/` 以下にソースコードを配置する
- `tests/` 以下にテストコードを配置する（対象ファイルと同じ階層構造）
- `docs/` 以下にドキュメントを配置する

## ディレクトリ構造

- ディレクトリは技術種別ではなく **役割・関心事** で切る（例: `domain` / `usecase` / `infra` / `ui`）
- 依存の向きを内側（domain）→外側（infra/ui）へ一方向に保つ。逆流させない
- 階層は 3〜4 段までを目安にする。深すぎる入れ子を作らない
- 新しいディレクトリは「既存のどれにも当てはまらない」ときだけ追加する
- 各ディレクトリの責務は README か index コメントで一言説明できる状態にする

## ファイル数・粒度

- 「1 ファイルの肥大化」と「小ファイルの乱立」の両方を避ける（目安 1 ファイル 400 行以内）
- 1 関数 / 1 型ごとにファイルを分割しない。凝集する単位でまとめる
- 迷ったら「関連するものは近くに（同一ファイル or 同一ディレクトリ）」を優先する

## インポート規約

- 相対パスより絶対パスを優先する（エイリアス設定済みの場合）
- 型のみのインポートは `import type` を使用する

## 関数設計

- 1関数 = 1責務を守る
- 引数は 3つ以下を目安にする（超える場合はオブジェクト化）
- 副作用のある関数と純粋な関数を分離する

## クラス設計

<!-- 設計レビューで繰り返し指摘されたパターンは、ここに追記して再発を防ぐ -->

- 生成方法が1つしかないクラスに named constructor（`fromString` 等）を作らない。
  素のコンストラクタで十分。named constructor は入口が複数ある / 生成の意味が異なる場合のみ
- 有限の値の集合（種別・ステータス等）は、言語が enum をサポートするなら enum で表現する
- DI で依存を受け取るだけのコンストラクタは設計ドキュメントに書かない（依存一覧の記載で足りる）。
  バリデーションや生成制約を持つコンストラクタは通常のメソッドと同様に書く

## コミットのタイミング

- 動作確認済みの単位でコミットする
- 途中状態（コンパイルエラーあり等）でコミットしない
- コミット前に、その言語のチェックを通す（例: TS 系 `npm run lint`（`--max-warnings 0`）+ `npm run typecheck` /
  Laravel `composer lint` + `composer analyse`）
