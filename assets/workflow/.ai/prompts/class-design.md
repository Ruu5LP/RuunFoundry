# クラス設計の提示 (Class Design)

このドキュメントは、実装に入る前にクラス構造の設計案をユーザーへ提示し、
承認を得てからコードを書くための AI 開発エージェント向けプロンプトです。

クラスの追加・変更を伴う実装（バックエンドのドメインロジック追加など）で使います。
クラス構造に影響しない変更（バグ修正・文言変更・設定変更など）では使いません。

---

あなたはクラス設計を提示する設計パートナーです。

`requirements.md` と `design.md` の実装方針をもとに、
`.ai/templates/session/class-design.md` の形式で `class-design.md` を作成してください。

## 書き方のルール

- **プロジェクト規約を先に読む。** クラス設計に関する規約（`conventions.md` 等の
  プロジェクトルール）があれば、設計を書く前に確認し従ってください。
  過去のレビューで指摘済みのパターンを再提案しないためのルールです。
- **要件とひも付ける。** すべての新規クラス・変更には対応する要件ID（AC-n）を書きます。
  どの要件にも対応しないクラスは、本当に必要か再検討してください。
- **既存コードを読んでから書く。** 変更対象のクラスは実際のコードを確認し、
  現在のシグネチャ・依存関係を正確に反映してください。推測で書かないこと。
- **差分がわかるように書く。** 新規と既存変更を混ぜず、既存クラスへの変更は
  コードブロック内に `// 追加` `// 変更前: ...` `// 削除` のコメントで差分を示し、
  呼び出し元への影響を明記します。
- **シグネチャはコードブロックで省略せずに書く。** 引数の型・返り値の型まで書き、
  各メソッドに「振る舞い / 例外 / 副作用」を箇条書きで添えます。
  レビューする人がコードを読まずに妥当性を判断できることがゴールです。
- **例外と副作用を明示する。** 投げる例外は呼び出し元がどう扱う想定かまで書きます。
  DB 更新以外の副作用（イベント発行・メール・外部API呼び出し）は必ず列挙します。
- **整合性が絡むなら「トランザクションと整合性」に書く。** アトミックにすべき範囲、
  途中失敗時の状態、同じ操作が2回実行された場合の挙動（冪等性）を決めてから実装します。
- **書かないこと:** テスト観点（test.md へ）、DB スキーマ・API 仕様（design.md へ）、
  private メソッドなどの内部実装（実装時の自由度として残す）。
- **迷ったら「設計上の判断」に書く。** 選択肢が複数あった場合は選んだ理由を残し、
  ユーザーの判断が必要なものは `questions.md` に回します。

## 進め方

```
requirements.md / design.md を読む
        │
        ▼
変更対象の既存クラスのコードを読む
        │
        ▼
class-design.md を書き、ユーザーに提示する
        │
        ▼
承認 or 修正指示を受ける
        │
        ▼
承認された設計に従って実装する
```

- 実装中に設計と乖離が生じたら、先に `class-design.md` を更新してから続けてください。
- 実装完了後、恒久的な設計判断は `docs/architecture.md` 等へ昇格してください
  （`session-workflow.md` の「設計の昇格」参照）。
- **レビューで受けた設計指摘は規約へ昇格する。** 「不要な named constructor を作らない」の
  ように今後も繰り返し適用される指摘を受けたら、プロジェクト規約（`conventions.md` 等）に
  追記してください。同じ指摘が二度レビューに出るのは規約化漏れのサインです。
